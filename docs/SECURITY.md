# Security and Authentication

The system-level STRIDE model, trust boundaries, automated independent review, residual risks, and human-review release requirement are maintained in `docs/THREAT_MODEL.md`. Vulnerabilities must be reported through the private process in the repository-root `SECURITY.md`.

## Local identity

Bootstrap runs at startup only when both `Bootstrap__AdminEmail` and `Bootstrap__AdminPassword` are present. It creates one active local user and the global system `Administrator` role only while the user table is empty. A database singleton records completion, survives administrator deletion, and makes later bootstrap attempts no-ops. Partial configuration, the documented placeholder password, and invalid credentials fail startup without logging either value. Remove both bootstrap settings, especially the password secret, from the runtime environment after the first successful start.

Local passwords are UTF-8 strings with a 1,024-byte ceiling; a bootstrap password must contain at least 12 characters. Passwords use Argon2id v19 with a random 16-byte salt, 32-byte output, `m=65536`, `t=3`, and `p=1`. Parameters remain in each PHC string, parsing accepts only the bounded current profile, comparisons are fixed-time, and unknown users run the same KDF through a dummy hash. One KDF runs at a time, consuming about 64 MiB of native memory; verification admits at most 16 active/waiting requests and waits at most two seconds. A future parameter change must allowlist the prior bounded profile until successful logins have rehashed it.

The browser receives independent 256-bit opaque access and refresh tokens. PostgreSQL stores only their SHA-256 hashes. Access tokens expire after 15 minutes by default; refresh families have an absolute 30-day lifetime that rotation never extends. Every refresh atomically revokes the presented session and rotates both tokens. A family-scoped PostgreSQL transaction lock orders rotation, replay response, and logout so a concurrently created replacement cannot escape revocation. Reuse of a replaced or revoked refresh token revokes the whole family. Revoked/replaced session predecessors and their token hashes are retained through the family's absolute expiry so replay remains detectable; cleanup may remove the chain only after that point. Logout revokes the family and is idempotent; login intentionally creates a new family, and refresh intentionally treats retries of a consumed token as replay.

`__Host-arrcontrol_session` and `__Host-arrcontrol_refresh` are `Secure`, `HttpOnly`, `SameSite=Lax`, `Path=/` cookies without a `Domain` attribute. Authentication mutations also require a 256-bit `__Host-arrcontrol_csrf` double-submit cookie with the same host/path constraints and an exact `X-CSRF-Token` header. The `__Host-` prefix prevents a sibling subdomain from setting a competing domain cookie. Obtain the CSRF token from `GET /api/v1/auth/csrf`; successful login and refresh rotate it and return the new value. There is no insecure-cookie override, so browser login requires HTTPS, including development behind a local TLS proxy.

Failed login responses do not distinguish unknown, disabled, or incorrect accounts. By default, five failed attempts per normalized account and 20 per client IP in 15 minutes are enforced from durable audit data before Argon2id. PostgreSQL transaction locks serialize each admitted account/IP check through its audit or successful session commit; an overlapping attempt is shed as rate-limited instead of overshooting the durable limit. Separate HTTP limits cap all login requests at 60 and all refresh/logout requests at 120 per peer IP in the same 15-minute window; an independent OIDC callback counter uses the latter limit so invalid protocol traffic cannot cause unbounded audit writes. These raw limits also cover invalid CSRF requests and do not replace the persistent failure limits. Limits and lifetimes are configured under `Auth__Local__*`. Audit actions cover bootstrap, login, refresh, refresh replay, and logout with correlation ID and IP, but never passwords, PHC strings, cookies, or raw tokens. TOTP and WebAuthn remain roadmap items.

## OIDC / Authentik

OIDC is disabled by default. Enabling it requires an exact HTTPS per-provider Authority ending in `/`, a confidential client ID/secret, and an origin-only HTTPS `App__PublicUrl`; placeholders and insecure or ambiguous URLs fail startup. The reference Authentik issuer is `https://auth.example.com/application/o/arrcontrol/`. Register strict typed redirect URIs `https://arrcontrol.example.com/auth/oidc/callback` for authorization and `https://arrcontrol.example.com/auth/oidc/signed-out` for logout. Reverse proxies must pass the scheme through explicitly trusted proxy addresses or networks. ArrControl also overwrites protocol redirect values with these configured exact URLs instead of deriving them from an untrusted Host header.

The handler uses a confidential Authorization Code client with PKCE S256, query response mode, scopes `openid profile email`, issuer discovery, correlation/state/nonce checks, and audience/lifetime validation. Authentik must have an asymmetric Signing Key and include claims in the ID token; ArrControl accepts RS256 and refreshes discovery keys after an unknown-key result. It does not call UserInfo and never retains provider access or refresh tokens. The validated ID token issuer must exactly equal the configured Authority; that issuer plus the token's single `sub` claim form a case-sensitive external key.

An existing `(iss, sub)` link wins even if later tokens omit or change email. A new identity is linked to the unique normalized local address, or provisions an active passwordless user, only when the ID token contains a syntactically valid email and boolean `email_verified=true`. Missing, false, malformed, inactive-user, unknown-role, oversized-claim, and protocol-failure cases fail generically without creating an identity or session. Authentik reports `email_verified=false` by default, so the reference realm must use a deliberate verified-email property mapping; never replace that mapping with an unconditional `true` claim.

Authentik `groups` values grant roles only through exact configured group-to-role mappings. Assignments are stored separately in `external_identity_roles` and fully reconciled on every successful login, so removed groups revoke only OIDC-owned grants and never overwrite manual `user_roles`. An OIDC-owned grant applies only to session families authenticated through that same current external identity: local sessions do not inherit it, and a session created through one linked external identity does not inherit another identity's grants. A configured target role that does not exist fails the login rather than silently dropping authorization. The Compose shortcut `ARRCONTROL_OIDC_ADMIN_GROUP` maps one exact group to the existing system `Administrator` role; indexed `Auth__Oidc__RoleMappings__N__Group` and `__Role` settings support other roles.

After all remote checks, ArrControl issues the same hashed opaque access/refresh tokens and Secure `__Host-` CSRF/session cookies used by local login. The session records `authentication_method=oidc`, which survives refresh rotation. Only the ID token needed as Authentik's RP-logout hint is retained, protected with ASP.NET Data Protection and associated with the local token family; it is never exposed as ArrControl application data or persisted raw. RP logout necessarily places the unprotected token in the browser redirect as the standard `id_token_hint`. OIDC logout first revokes the complete local family and deletes cookies, then redirects to the discovered end-session endpoint with that hint and the exact allowlisted post-logout URI. Loss of the provider or key ring cannot undo the local logout.

Data Protection state contains the PKCE verifier and protects the logout hint. Compose mounts a writable non-root key directory while the rest of the container remains read-only. A manually persisted filesystem key ring is not automatically encrypted at rest: restrict and back up the Docker volume like a credential, and use a protected shared key repository before running multiple replicas. Losing the key ring invalidates in-flight OIDC callbacks and provider logout hints, but not ArrControl's opaque local sessions.

Provider-side account disable or sign-out does not yet revoke an already issued ArrControl family; it remains valid until local logout, replay revocation, user disable, or the configured absolute refresh expiry. Authentik back-channel logout is a separate future hardening slice. Local password login therefore remains available; it may be disabled only after a tested OIDC Administrator mapping and documented recovery path, and no disable switch is exposed yet.

## Authorization permissions

The immutable permission catalog is `instances.read`, `instances.manage`, `library.read`, `search.execute`, `queue.manage`, `tasks.execute`, `users.manage`, `audit.read`, `settings.manage`, and `authorization.manage`. The built-in system-role bundles are:

| System role | Permissions |
|---|---|
| Administrator | all ten permissions |
| Operator | `instances.read`, `library.read`, `search.execute`, `queue.manage`, `tasks.execute` |
| Viewer | `instances.read`, `library.read` |

System roles cannot be changed or deleted. A custom role is an administrator-defined bundle drawn only from the catalog. Manual role assignments apply either globally or to exactly one instance group. Effective grants are the union of all applicable assignments; there are no deny rules, and a global grant dominates and suppresses redundant group scopes. A group-scoped permission applies only when the command or query targets that group. An ungrouped instance is accessible only through a global grant, and an endpoint without an instance-group resource, including authorization administration, always requires a global grant.

Manual `user_roles` assignments apply to every valid session for their user. OIDC-owned roles are global but apply only to a session family associated with the same current `external_identity`, as described above. Effective authorization is resolved from current database state on each request rather than embedded in the opaque session, so assignment and role changes take effect without issuing new cookies. Permissions are checked in application commands and filtered in queries; hiding a button is not authorization. In particular, `GET /api/v1/instances` requires `instances.read`, returns `403 access_denied` when the user has no grant in any scope, and performs group filtering in SQL before materialization.

`GET /api/v1/auth/me` is available to every authenticated user and returns their effective permission codes with either a global marker or the sorted set of allowed instance-group IDs. The permission catalog, roles, users, manual assignments, and available instance groups are administered under `/api/v1/authorization`. Every endpoint in that group requires a global `authorization.manage` grant. `PUT /authorization/roles/{roleId}` is an idempotent create-or-replace operation for custom roles; `PUT /authorization/users/{userId}/role-assignments` idempotently replaces the complete manual assignment set and never changes OIDC-owned assignments. Authorization `PUT` and `DELETE` requests require the same exact double-submit CSRF cookie/header pair used by authentication mutations.

Custom-role upserts/deletes and manual-assignment replacements are audited with actor, target, scope, correlation ID, a redacted summary, and outcome. They are rejected when they would leave no active user with a global `authorization.manage` grant. Authorization failures use stable RFC 9457 codes: malformed input uses `invalid_request`, while validated authorization input uses `role_id_invalid`, `role_name_invalid`, `role_permissions_invalid`, `user_id_invalid`, or `role_assignments_invalid` (400); authentication uses `authentication_required` (401); authorization or CSRF denial uses `access_denied` or `csrf_validation_failed` (403); missing references use `role_not_found`, `user_not_found`, or `instance_group_not_found` (404); and conflicts use `role_name_conflict`, `system_role_immutable`, or `authorization_lockout` (409).

## Secrets and transport

Provider/API credentials use AES-256-GCM with a versioned 32-byte master key mounted from an absolute filesystem path. A key file contains exactly the canonical Base64 encoding of 32 random bytes, optionally followed by one line ending. Startup rejects partial settings, non-positive or duplicate versions, relative/unreadable paths, malformed Base64, non-32-byte keys, or an active version absent from the configured key set; error and object text never include a key or path. With no key settings, read-only startup remains possible but credential writes and provider reads fail closed.

Each encryption uses a fresh 96-bit nonce and a 128-bit tag. Authenticated associated data binds the format version, instance ID, exact credential purpose, and key version, so a database value cannot be moved to another instance or purpose undetected. Ciphertext, nonce, tag, and key version live in PostgreSQL. The browser API accepts only the known `api-key`, `username`, and `password` purposes and never exposes plaintext, ciphertext, nonce, tag, or key version: listing and mutation responses contain only purpose, `configured: true`, and update time. Each adapter declares its exact required purposes, and scheduled resolution decrypts only credentials belonging to that instance into an ephemeral redacted connection object. The only cleartext read boundary is the internal provider service. Credential requests, domain objects, audit rows, validation errors, and logs are redacted.

The database has no plaintext credential column and enforces a non-empty ciphertext, a 96-bit GCM nonce, a 128-bit authentication tag, and a positive key version. Encryption and redaction still happen before persistence; schema constraints are defense in depth, not a substitute for the credential-store boundary. Credential writes require effective `instances.manage`, metadata reads require `instances.read`, and both enforce the target instance-group scope. Mutations require CSRF validation and emit redacted audit events. Audit rows retain a non-secret actor snapshot, are time-partitioned, and cannot be updated in place. TLS verification defaults on. CSP, HSTS, anti-forgery, secure headers, request-size limits, and dependency scanning are release gates.

## Threat considerations

Instance base URLs must be absolute HTTP(S) URLs without user information, query, or fragment and are normalized with a trailing slash. Creation and update resolve every DNS answer and fail closed if resolution fails. Loopback, unspecified, link-local/cloud-metadata, carrier-grade NAT, multicast, documentation/benchmark/reserved, and equivalent embedded NAT64 destinations are always blocked. RFC1918 and IPv6 ULA/site-local destinations require the per-instance `allowPrivateNetworkAccess` opt-in, and a mixed public/private DNS answer is rejected unless every address is permitted.

Every outbound connection re-resolves the hostname immediately before use, rejects the complete answer set again, and connects through the validated IP list while retaining the original host for HTTP Host and TLS SNI/certificate validation. The generic connection probe has a ten-second bound, disables proxies, cookies, decompression, and redirects, reads response headers only, and never sends a credential; any HTTP response is transport connectivity rather than provider compatibility. TLS verification defaults on. Turning it off is stored and returned as an explicit high-risk instance property, never an automatic fallback.

The Arr library adapters use the same address-pinned boundary with a 15-second per-request bound and a 32 MiB response limit suitable for bounded library snapshots. They send the decrypted API key only in the official `X-Api-Key` request header, never in a URL, object text, result, exception, audit event, or log. Adapter-authored catalog query keys are ASCII-allowlisted and values are URI-escaped; credentials are forbidden in query parameters. Calls independently re-resolve and pin DNS, do not use environment proxies, do not follow redirects, do not retain cookies, and accept only the expected provider name plus allowlisted contract-tested major versions (Sonarr 4; Radarr 5/6; Lidarr 2/3; Readarr 0.4; Whisparr 2/3). Missing/tampered/unavailable credentials fail as the non-secret `credential_missing` outcome. Authentication, authorization, rate limiting, version, response, timeout, TLS, and network failures map to stable non-secret codes. Response buffers are cleared after typed parsing.

Prowlarr and Bazarr use the same pinned transport and header-only key handling. Prowlarr indexer DTOs discard provider configuration fields, endpoints, and upstream response data before leaving the adapter. Bazarr's health `object` field may be a filesystem path or private profile name; it is never returned or persisted and contributes only to a one-way SHA-256-derived source identifier. Bazarr status parsing ignores directory/configuration fields, and subtitle activity retains only daily aggregate counts. The supported allowlists are Prowlarr major 2 and Bazarr major 1.

Download-client adapters use the same DNS-rebinding-resistant transport, response/body limits, redirect ban, proxy ban, and stable errors. NZBGet and Transmission construct Basic authorization headers from per-instance username/password values; qBittorrent and Deluge submit credentials only to their official login RPCs and retain a validated, bounded session cookie for the immediately following calls. Transmission accepts only the bounded CSRF header returned by a 409 challenge. SABnzbd's official API requires an `apikey` query parameter, so it is the single explicit exception to header/body credential transport; the transport and adapter never render request queries, headers, bodies, cookies, or URIs into object text, exceptions, audits, or application logs. Request/login buffers and all response buffers are zeroed after use. Supported allowlists are SABnzbd majors 4/5, NZBGet majors 25/26, qBittorrent major 5, Transmission major 4, and Deluge major 2.

Media-server adapters send API keys only in `X-Plex-Token` or `X-Emby-Token` headers through the same pinned transport. They never retain or expose account/user names, server/library names, playback titles, device/client identifiers supplied by the server, client addresses, or filesystem/transcode paths. Plex's required client identifier is derived from the ArrControl instance UUID and is not a credential. Only bounded aggregate counts leave the adapter. Supported allowlists are Plex 1.41–1.43, Jellyfin 10.10/10.11, and Emby 4.8/4.9; unknown future versions fail closed.

Request-manager adapters use the same pinned transport and send API keys only in the upstream-defined `X-API-Key` (Overseerr/Jellyseerr/Seerr) or `ApiKey` (Ombi) header. Provider responses are bounded to 10,000 requests and discard titles, users, aliases, rejection reasons, root folders, and raw payloads before returning. Supported allowlists are Overseerr 1.34/1.35, Jellyseerr 2.7/Seerr 3.0–3.3, and Ombi 4.47/4.48; unknown future versions fail closed.

Recyclarr is never treated as an HTTP target. Its CLI adapter accepts absolute executable/configuration paths and only constructs fixed `--version` or `sync` argument lists with `sonarr|radarr`, validated alphanumeric/underscore instance names, optional `--preview`, and non-interactive logging. It never invokes a shell. The child environment is cleared before adding only `NO_COLOR`, `RECYCLARR_CONFIG_DIR`, and `RECYCLARR_DATA_DIR`, preventing ArrControl secrets from being inherited. Output is bounded and redacted from result text; cancellation/timeout terminates the complete child process tree. Only allowlisted major versions 7/8 execute sync, and live mutations remain forbidden without an explicit caller action.

HTTP notification adapters use the address-pinned provider transport: redirects, proxies, cookies, environment credentials, and DNS rebinding are blocked. Endpoint URLs, query values, webhook path tokens, headers, JSON bodies, SMTP addresses, usernames, passwords, bot tokens, user keys, and provider response bodies are redacted from object text and stable errors. Generic webhook signatures use HMAC-SHA-256 over the exact body. Provider payloads disable Discord mentions and omit markup/interactive actions. Telegram, Pushover, and Gotify validate their documented success response or status. Email accepts only `smtps://` with certificate verification enabled; SMTP connects to a policy-approved IP while authenticating TLS against the original hostname, authenticates only inside TLS, bounds multiline replies, sanitizes addresses, encodes subjects, dot-stuffs bodies, and clears the temporary AUTH buffer. No notification is sent automatically by adapter registration.

Catalog persistence contains normalized media fields, source identifiers, fingerprints, and an adapter-selected non-secret provider-data projection; it does not retain raw response bodies, filesystem paths, API keys, or authorization headers. A maximum of 250,000 normalized items per instance snapshot limits memory/database amplification. Duplicate keys, orphan parents, invalid bounds, malformed JSON, or incomplete provider calls reject the whole snapshot before existing rows are removed.

Missing reads require the current effective `library.read` grant and apply global/instance-group visibility in SQL before filters or pagination. The opaque cursor is protected by the shared ASP.NET Core Data Protection key ring, expires operationally when that key material is rotated away, is capped at 4 KiB, and is cryptographically bound to the normalized filter. Cursor contents cannot broaden RBAC scope. Saved views are keyed to the authenticated user; another user's UUID returns not found. Their persisted filter may intentionally contain a title search, but audit scope/summary records only the view UUID and mutation outcome. PUT/DELETE require the normal exact CSRF cookie/header pair.

Queue/history reads require current `instances.read` scope and apply it in SQL. Activity adapters discard output paths, error messages, and provider data dictionaries; only bounded operational fields are persisted. Download identifiers are never logged or returned by the aggregation API. Correlation uses a one-way SHA-256 key over a case-normalized ID within the same instance, preventing title-based false matches and cross-instance joins. Snapshot size/page bounds prevent unbounded upstream or database amplification.

Import-failure classification never parses titles, paths, raw error messages, or other free text and never uses probabilistic/AI inference. Exact fail-soft rules over allowlisted normalized fields produce stable evidence IDs. This avoids both sensitive-text processing and a misleading repair diagnosis. No rule enables retry, filesystem changes, or arbitrary commands.

Operation idempotency keys and request hashes are validation/control data: neither is returned, audited, or logged. Creation serializes the actor/route/key tuple with a PostgreSQL advisory transaction lock, preventing concurrent duplicate target snapshots. Per-target result JSON is provider-independent, valid JSON, and limited to 64 KiB; callers/handlers must never place credentials or raw upstream bodies in it. Cancellation requires authentication, ownership, and CSRF. Aggregate and target state constraints reject invalid lifecycle values at the database boundary.

Search preview and start both recompute eligibility under the current `search.execute` global/group grants; client-supplied UUIDs never become operation targets directly. Non-visible selections contribute only to an excluded count, avoiding existence disclosure. The only provider mutation payloads are adapter-constructed `EpisodeSearch`/`MoviesSearch` JSON with positive integer IDs read from normalized provider keys, a maximum batch of 100, and a 1 MiB transport body cap. The body buffer is zeroed after the request, credentials remain header-only, and upstream response/exception bodies are not persisted.

Health incidents use a SHA-256 grouping key over provider kind plus a normalized source or safe remediation URI; diagnostic message text never affects identity and therefore cannot manipulate grouping. Raw source messages are bounded to 4,000 characters, available only through the scoped health read API, and excluded from logs and audit summaries. Provider remediation links are allowlisted to absolute HTTP(S) URLs before persistence and the web client opens them as untrusted external navigation with `noreferrer` behavior. Acknowledge/snooze authorization is recomputed against the incident's current instance-group scope before each write; snooze duration is capped at 30 days.

Outbox events are invalidations, not data replication: their payload allowlist contains resource/permission names and UUID scope metadata only. Publication never serializes changed entities, provider diagnostics, media metadata, audit content, credentials, or operation results. SSE authentication uses the normal secure session cookie. Stored target UUIDs are intersected with current RBAC and current instance-group membership before emission; actor-owned events require an exact current user match. Cursor IDs grant no access, are validated against published retained rows, and deliberately force snapshot recovery after expiry. Proxy buffering and response transforms are disabled for the stream, while heartbeat/event output remains bounded.

Complete audit reads and diagnostics exports require a global `audit.read` grant; a scoped grant is deliberately insufficient because authentication and system events have no honest instance boundary. Audit cursors are data-protected and bound to the normalized filter. Diagnostics generation uses an explicit allowlist rather than recursive object redaction: it never reads credential rows, provider/media payload fields, instance names/URLs, health messages, audit actor identifiers/IPs/summaries/correlation IDs, or stable instance/group UUIDs into the archive. Per-export references prevent cross-bundle correlation. The export is CSRF-protected, size/time bounded, `no-store`, and audited with only lookback/include-audit settings.

Instance reads and mutations enforce current global/instance-group RBAC. Moving an instance requires `instances.manage` in both scopes; group administration requires global `instances.manage`. Mutations require CSRF and emit audit summaries without base URLs, DNS answers, credentials, or response bodies. Logs redact authorization headers, API keys, cookies, query secrets, filesystem paths, and media names in support bundles unless opted in.

All GitHub Actions dependencies are pinned to immutable full commit identifiers and grouped Dependabot updates keep those pins reviewable. This is especially important for third-party scanners and signing installers. CI runs CodeQL and a commit-pinned Trivy release; tag builds sign only immutable image digests with short-lived GitHub OIDC credentials, then verify the expected workflow identity. No long-lived Cosign private key is stored in repository secrets.

Job leases use an unpredictable per-claim UUID in addition to owner and expiry. Renewal, failure, completion, abandonment, and checkpoint advancement compare that token, so a timed-out or partitioned worker cannot commit after another replica reclaims the row. Scope JSON, cursors, and stable error codes must not contain credentials or raw provider payloads. Global concurrency, handler timeouts, capped attempts/backoff, materialization bounds, and database constraints limit resource amplification; provider-specific rate limits remain the responsibility of each adapter/handler boundary.

Security reports go through GitHub private vulnerability reporting. Never open a public issue containing a live credential or exploit details.
