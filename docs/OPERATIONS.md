# Deployment, Operations, and CI/CD

Use `ADMIN_GUIDE.md` for the end-to-end administrator workflow, `OPERATOR_GUIDE.md` for incident handling, and `PROVIDER_TROUBLESHOOTING.md` for stable provider outcomes. This document is the lower-level deployment reference.

## Images and tags

GitHub Actions builds `nobentie/arrcontrol` on Docker Hub and `ghcr.io/nbt4/arrcontrol`. Pull requests and `main` build/test without publishing; semantic tags publish the exact version, major/minor aliases, and `latest` only for stable releases. Release images target linux/amd64 and linux/arm64, run non-root, include OCI labels plus BuildKit SBOM/max-level provenance attestations. Workflow actions are pinned to full commits and updated by Dependabot.

The tag must exactly match `Directory.Build.props`, and publication is blocked until `docs/release/SECURITY_REVIEW.md` records independent approval. The workflow asserts amd64/arm64 entries after push and retains the manifest, compatibility report, digest, and signature verification as release evidence.

The CI image gate loads a single-architecture image locally, proves its configured non-root user, starts it with a read-only root filesystem, and exercises liveness, public status, and the embedded SPA. Trivy blocks fixed high/critical image vulnerabilities and generates a retained CycloneDX SBOM; its repository scan also covers vulnerable dependencies, committed secrets, and configuration errors. CodeQL analyzes C# and TypeScript on pull requests, `main`, and weekly. Release publication repeats the high/critical scan before signing the immutable digest and verifies the resulting certificate identity. A failed scan or signature check leaves tags potentially uploaded but does not produce a verified release; remove any such registry tags before retrying.

Verify a tagged image independently by digest, substituting the release tag and repository path:

```text
docker pull nobentie/arrcontrol:1.0.0
cosign verify \
  --certificate-identity-regexp '^https://github.com/nbt4/ArrControl/.github/workflows/release.yml@refs/tags/v' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  ghcr.io/nbt4/arrcontrol@sha256:...
```

## Compose lifecycle

Copy the root `.env.example` to `.env`, rotate secrets, and create the credential key with `umask 077; mkdir -p secrets; openssl rand -base64 32 > secrets/master-key`. Run `docker compose up -d`. ArrControl waits for PostgreSQL health, applies pending migrations under a PostgreSQL advisory lock, and starts HTTP/background services only after migration succeeds. Pin `ARRCONTROL_IMAGE` to a version in production, not `latest`. Keep the key file owned by the image's runtime UID/GID with mode `0600`; Compose mounts it read-only at `/run/secrets/arrcontrol_master_key`. The complete tested backup, restore, validation, and rollback procedure is in `docs/BACKUP_RESTORE.md`.

Baseline Compose reserves 256 MiB of shared memory for PostgreSQL so parallel projection queries do not exhaust Docker's small default `/dev/shm`. The validated data/concurrency ceiling and the measurements that support it are in `docs/PERFORMANCE.md`; benchmark the actual host before exceeding that envelope.

For a controlled upgrade, stop `arrcontrol`, take and verify a database backup, pull the pinned target image, then run `docker compose up -d`. ArrControl applies pending migrations before its HTTP server starts; inspect `docker compose logs arrcontrol` if startup fails. The same image can run outside Compose as `ArrControl.Api database migrate` with only `ConnectionStrings__Database` configured. The command serializes competing migrators with a five-minute-bounded PostgreSQL advisory lock, logs only migration counts, and is safe to rerun. It does not bootstrap users, start HTTP listeners, or require provider credential keys. Never grant the normal application database account less schema authority than the migration command unless a separate migration connection setting is introduced and documented first.

`ARRCONTROL_MASTER_KEY_VERSION` selects the version used for new writes. Rotation is additive: mount the new key as another indexed `CredentialEncryption__Keys__N__Version`/`__Path` pair, keep every old version required by existing rows, switch `CredentialEncryption__ActiveKeyVersion`, restart, and rewrite credentials through the API as operationally convenient. Never reuse a version for different bytes and never remove an old key while PostgreSQL still references it. Loss of an old key makes those credentials undecryptable; restoring only the database is insufficient.

## Reverse proxy

Terminate TLS at Caddy, Traefik, or nginx. Preserve WebSocket/SSE upgrades and request IDs. Configure trusted proxy networks; do not trust arbitrary forwarded headers. Public URL and OIDC redirect origin must match exactly.

Local-session cookies use the `__Host-` prefix and are always `Secure`, host-only, and `Path=/`; exposing the application directly over plain HTTP does not provide a working browser login. Client-IP authentication throttling uses the direct peer unless at least one trusted `ReverseProxy__KnownProxies__N` IP or `ReverseProxy__KnownNetworks__N` CIDR is configured; `ReverseProxy__ForwardLimit` defaults to one and is bounded from one through ten. With no trust entries, forwarded headers remain disabled. Never add an arbitrary-source network, because spoofed client addresses bypass IP controls.

OIDC additionally requires `ARRCONTROL_PUBLIC_URL` to be the externally visible HTTPS origin and the Authentik Authority to use its exact per-provider issuer path. Register the authorization and logout callback URLs printed in `docs/SECURITY.md` as strict typed Authentik URIs. Use a confidential Authorization Code provider, grant types limited to authorization code, include claims in the ID token, and select an RSA Signing Key that emits RS256 tokens. Set `ARRCONTROL_OIDC_ENABLED=true`, the Authority/client credentials, and optionally the exact `ARRCONTROL_OIDC_ADMIN_GROUP`; startup fails before discovery if this configuration is unsafe. Do not request `offline_access`.

The `data-protection-keys` volume is the only writable persistent application path in the baseline Compose service. Back it up with restricted access alongside the database, because it protects OIDC state/PKCE and the stored logout hint. The filesystem key ring is not automatically encrypted at rest. A restored database without the matching key ring keeps local sessions usable but makes in-flight callbacks and old upstream logout hints unusable; local revocation still works. Multiple API replicas require one protected shared key ring and are not supported by the single-host named-volume baseline.

## Local administrator bootstrap

Run schema migrations before the first application start (the baseline Compose dependency does this automatically). Supply both bootstrap variables for exactly one app start, confirm the generic completion log entry, verify login over HTTPS, and then remove both variables from Compose and the process environment and delete the password from the secret store. The database bootstrap sentinel prevents a later restart or administrator deletion from re-enabling bootstrap. A partial configuration, the placeholder password, invalid local-auth limits, or an unavailable/missing schema causes startup to fail closed.

Monitor `identity.login` failures/rate limits and `identity.refresh_reuse` events. A refresh replay revokes its entire token family; repeated events can indicate a stolen cookie or a client retrying a consumed refresh after losing the successful response.

Monitor `identity.oidc_login` outcomes separately. `unverified_identity`, `inactive`, and `role_missing` are policy failures; repeated protocol failures can indicate stale redirect/client configuration, clock skew, or malicious callbacks. Audit summaries contain only `method: oidc`, never codes, state, nonce, provider tokens, claims, or the client secret.

## Instance onboarding and probes

Create instance groups before assigning instances. Group creation/rename/deletion requires a global `instances.manage` grant; a group with instances cannot be deleted. Instance creation and updates resolve the configured hostname during validation, so DNS must be available even when the upstream application is temporarily offline. Homelab RFC1918/ULA targets require the explicit private-network flag. Loopback, link-local/metadata, CGNAT, multicast, reserved, and mixed unsafe DNS answers remain blocked even when private access is enabled.

Run the connection probe after onboarding and after transport changes. Contract-supported adapters perform typed status/health validation and expose provider version, normalized health issues, capabilities, and rate-limit metadata without exposing credentials. Arr-family, Prowlarr, Bazarr, SABnzbd, Plex, Jellyfin, Emby, Overseerr, Jellyseerr, and Ombi require `api-key`; NZBGet, qBittorrent, and Transmission require `username` plus `password`; Deluge requires `password`. An Emby instance base URL is the server root, because the adapter appends its documented `/emby` API prefix. Other configured kinds receive only a credential-free generic transport check and are not thereby supported. `credential_missing`, `unauthorized`, `forbidden`, `rate_limited`, `unsupported_version`, `invalid_response`, `timeout`, `tls_error`, and `unreachable` are stable adapter outcomes. A generic `connected` result means only that an HTTP response arrived. TLS verification should remain enabled; disabling it is a deliberate per-instance exception and must be documented operationally. Changing kind, URL, TLS verification, or private-network policy clears stale capability observations until the next probe.

## RBAC administration and recovery

The RBAC migration installs the immutable Administrator, Operator, and Viewer bundles plus the canonical permission catalog. After upgrading, verify that at least one active account has a global Administrator or other role containing `authorization.manage` before delegating scoped access. The settings client may read `/api/v1/authorization/permissions`, roles, users, and instance groups, then use idempotent `PUT` requests with the normal CSRF cookie/header pair to manage custom roles and manual assignments.

ArrControl serializes authorization mutations and rejects a change that would remove the last active global authorization manager. OIDC-owned grants count for recovery only while a matching OIDC session family is currently valid; keep the tested local bootstrap administrator as the independent recovery path. Monitor `authorization.role_upsert`, `authorization.role_delete`, and `authorization.assignments_replace` audit outcomes. Their summaries contain only target identifiers, counts, or stable error codes.

The default raw per-peer limits are 60 login requests and 120 combined refresh/logout requests per 15 minutes, independent of the durable five-per-account and 20-per-IP failed-login limits. OIDC callback and signed-out traffic has a separate counter capped by the same 120-request setting, preventing invalid callbacks from producing unbounded audit writes. Tune `Auth__Local__LoginRequestLimit` and `Auth__Local__SessionMutationRequestLimit` only with memory capacity and trusted-proxy behavior in mind. Retain replaced/revoked session-chain rows until the refresh family's absolute expiry; deleting them earlier disables replay detection for consumed tokens.

Budget at least 64 MiB of native memory above the normal application/database-client working set for local password verification. ArrControl serializes Argon2id work and rejects excess or more-than-two-second verification waits instead of increasing KDF memory pressure.

## Observability

`/health/live` checks process liveness; `/health/ready` checks PostgreSQL connectivity and reports unhealthy while any EF Core migration is pending. The HTTP process never applies migrations automatically; only the explicit command/Compose one-shot does. OpenTelemetry OTLP export is optional. Logs are JSON in production. Recommended alerts: readiness failure, poll success age, job backlog, provider error ratio, database saturation, and repeated authentication failures.

The embedded automation worker is leaderless: every replica may plan and claim work. Schedule cursors prevent duplicate materialization, `FOR UPDATE SKIP LOCKED` distributes due rows, and a random lease token fences a worker after expiration/reclaim. Handlers renew at one third of the 30-second default lease and time out after two minutes. Failures retry up to five attempts with capped exponential backoff and deterministic jitter; process shutdown returns owned work to `retry`, while exhausted expired leases become terminal `lease_expired`. Successful job state and all returned sync checkpoints commit in one transaction.

Tune `ARRCONTROL_SCHEDULER_MAX_CONCURRENCY` only against PostgreSQL and upstream capacity; the allowed range is 1–64. `ARRCONTROL_SCHEDULER_MAX_ATTEMPTS` is 1–100. Further duration settings are available under `Automation__Scheduler__*` and validate against documented safe bounds at startup. Alert on terminal `failed` jobs, repeated claim/planning warnings, lease expiry, and old `sync_checkpoints.last_success_at`. Do not repair leases by hand while workers run; disable the schedule, let the lease expire, diagnose the stable error code, then re-enable or create a deliberate replacement run.

Every enabled contract-supported Arr library instance is reconciled to one UTC `catalog.sync` schedule (`*/15 * * * *`), matching the SRS library freshness target. The instance must have a decryptable `api-key` credential. The worker reads only official catalog endpoints, retains the prior projection on any partial/error response, and records stable `catalog_*` job error codes without upstream bodies or media titles. Disabling/deleting the instance or changing it to another kind disables the schedule within one minute. Monitor the per-instance `catalog` checkpoint; its JSON cursor records strategy, observation time, snapshot hash, and item count but no media content or credential.

`GET /api/v1/missing` reads only PostgreSQL and never blocks on an upstream. It defaults to currently available missing movies, episodes, albums, and books; add `reason=not_available` to include monitored future releases/airings. Repeated `instanceId`, `kind`, and `reason` parameters, plus a case-insensitive literal title search, are supported. Pages use a protected keyset cursor bound to the normalized filter; changing a filter requires starting without the old cursor. Every eligible visible source appears in `freshness`, including sources with zero results. `stale` becomes true after 30 minutes without a successful catalog checkpoint. Saved-view mutations are user-owned, CSRF-protected, idempotent by UUID, audited, and case-insensitively name-unique.

Enabled contract-supported Arr library, Prowlarr, SABnzbd, NZBGet, qBittorrent, Transmission, and Deluge instances receive a 30-second `activity.sync` schedule. Each successful pass atomically replaces the current queue, upserts at most 10,000 newest history records, deletes history older than 30 days, and advances the `activity` checkpoint. `/api/v1/queue` and `/api/v1/history` aggregate only currently visible `instances.read` scopes and expose correlation counts/flags computed from normalized download IDs. Rows older than two minutes are marked stale. The download-client adapter boundary supports pause/resume and remove for all five clients, plus retry for SABnzbd and NZBGet; a public RBAC/audit-controlled queue mutation endpoint remains a separate backlog slice, so the current HTTP aggregation API is still read-only.

Prowlarr queue is empty and its exact-time indexer events appear in aggregated history without query terms or response data. All contract-supported Arr, supporting, download-client, media-server, and request-manager adapters participate in the five-minute health schedule. Bazarr health source objects are hashed before persistence. Its subtitle-activity adapter exposes exact daily counts only; it does not reinterpret locale-relative detail-history timestamps. Torrent history contains only completed torrents still returned by the client's current list; removed torrents cannot be reconstructed. Media-server probes validate their aggregate availability/playback endpoints; no separate activity schedule or playback-history persistence is created. Request-manager health reads validate their authenticated bounded inventory but do not yet persist request rows.

Recyclarr is a local CLI integration, not an instance URL. Operators who embed it in a native ArrControl deployment must install a trusted Recyclarr 7/8 executable and keep its configuration directory outside the web root with restrictive permissions. The adapter does not inherit the ArrControl environment and therefore expects Recyclarr secrets/configuration in that dedicated directory. Always run preview first; a non-preview sync changes Sonarr/Radarr configuration. The baseline ArrControl container intentionally does not bundle or mount another project's executable, and no live sync runs automatically.

Notification destinations are source-level outbound adapters in this wave; registration alone does not create routing or contact a destination. Treat Discord/Slack/Teams webhook URLs, Telegram bot tokens, ntfy access tokens, Gotify application tokens, Pushover application/user keys, generic signing secrets, and SMTP credentials as write-only secrets. Require HTTPS for hosted HTTP services and `smtps://` with verified certificates for email. Self-hosted ntfy/Gotify/private SMTP targets require the same explicit private-network authorization as provider instances. Teams destinations must use a current Teams Workflows incoming-webhook URL; legacy Office 365 Connector URLs are not the tested contract.

Import triage is deterministic. Ordered rules inspect only normalized queue `status`, `trackedStatus`, `trackedState`, or history `eventType` and return category, severity, stable rule ID, and a localized guidance key. Unknown combinations return no classification rather than a guess. `retrySupported` remains false for every rule because no safe retry-import contract has been established; use the provider guidance/link-out path instead.

Enabled Sonarr/Radarr instances receive a five-minute `health.sync` schedule. A successful poll groups issues with the same provider remediation URI, or the same normalized source when no safe HTTP(S) URI exists, while retaining each bounded source diagnostic and its first/last sighting. An empty successful snapshot resolves active incidents; a later recurrence reopens the same stable incident ID and clears stale acknowledgement/snooze state. Failed polls retain the prior incident projection and surface through the durable job error/checkpoint path. Active incidents older than ten minutes are marked stale.

`GET /api/v1/health/incidents` applies `instances.read` scopes in SQL and excludes resolved incidents by default. Acknowledge and snooze mutations require CSRF plus `tasks.execute` for the incident's current instance-group scope; ungrouped instances require a global grant. Snoozes must be in the future and are capped at 30 days. Mutation audits contain identifiers, booleans, and the bounded snooze timestamp only—never the raw health message. Remediation links are emitted only when the adapter parsed an absolute HTTP(S) URI.

Broad commands use the durable operation model. Clients supply a non-secret `Idempotency-Key`; the server binds it to actor and route for 24 hours and compares a canonical SHA-256 request hash. An identical retry returns the original operation, while changed input returns `409`. `dryRun` is persisted on the aggregate and must be honored by each operation handler. Target UUID/key pairs are snapshotted before execution and retain independent succeeded/failed/skipped/cancelled outcomes; the aggregate resolves to succeeded, failed, partial, or cancelled. `DELETE /api/v1/operations/{id}` requests cooperative cancellation and immediately cancels an operation that has not started. GET/cancel is actor-owned and does not disclose another user's operation.

Operational EF writes append compact invalidation messages to `outbox_messages` in the same database commit. Messages contain only a resource name, required permission, actor UUID where ownership applies, and instance/group UUID targets—never provider payloads, titles, paths, credentials, audit summaries, or operation results. Raw SQL operation-progress writes add their outbox message explicitly inside the same transaction. A leaderless publisher claims at most 200 unpublished rows with `FOR UPDATE SKIP LOCKED`, marks them visible, and retries by leaving a failed batch unpublished. Published rows are retained for seven days and deleted in bounded hourly batches.

Live clients first call `GET /api/v1/events/snapshot`, then read the listed REST projections, and finally open `/api/v1/events?cursor=...`. This order closes the race between snapshot and stream. The SSE channel replays published rows after that waterline, emits cursor-only advancement for events hidden by RBAC, uses `Last-Event-ID` on browser reconnect, and sends an explicit `snapshot-required` event before closing when a cursor is missing, malformed, or older than retention. The client must repeat the snapshot sequence in that case. Idle connections receive 15-second heartbeat comments. Every event is re-filtered against current grants and current instance groups; multiple replicas coordinate only through PostgreSQL and may serve any reconnect.

`GET /api/v1/audit` requires global `audit.read`, because authentication/system events do not have a safe instance scope. It defaults to the latest 30 days, permits exact actor/action/outcome/correlation and bounded time filters, and uses a protected filter-bound descending keyset cursor. Scope and summary JSON are the already-redacted structures written by each mutation boundary; audit readers should still treat actor identifiers and processed client IPs as sensitive operational data.

Audit retention defaults to 365 days and runs daily at 03:15 UTC as durable `audit.retention` work. `ARRCONTROL_AUDIT_RETENTION_DAYS` is bounded to 30–3,650 days; batch size and maximum batches are also bounded. Each run deletes oldest rows in limited batches inside one transaction, then appends a system audit event with cutoff/count/limit status. A limit-reaching run reports `partial` and the next daily run continues. The database update trigger continues to reject in-place audit modification; retention uses deletion only.

`POST /api/v1/diagnostics/export` is an explicit, CSRF-protected global `audit.read` action and is itself audited. Its ZIP contains one `diagnostics.json` with pseudonymous per-export instance/group references, provider kinds and safe policy flags/capabilities, aggregate job error codes, checkpoint timestamps, aggregate health counts, and optionally up to 10,000 recent audit action/outcome/scope-kind rows. The strict profile excludes credentials, names, email addresses, IPs, URLs, paths, media metadata, raw provider fields, audit summaries/correlation IDs, and stable instance/group UUIDs. Lookback is 1–168 hours; repeated exports are intentionally distinct audited reads and do not use idempotency keys.

Search always begins with `POST /api/v1/operations/search/preview`. Scope mode is exactly one of selected media UUIDs (up to 10,000), instance UUIDs, group UUIDs, or all; resolution intersects current `search.execute` scope, enabled contract-tested instances, and the current available-missing projection. The preview reports excluded input count, per-instance counts, and exact provider target keys. Starting the operation repeats resolution, snapshots targets, and requires CSRF plus `Idempotency-Key`. Dry-run targets never call an upstream. Live execution batches at most 100 provider IDs, serializes calls per operation, spaces calls to an instance by at least two seconds, honors bounded Retry-After delay metadata, checks cancellation between instance groups, and records stable target errors without aborting other instances.

## Release gates

Backend/frontend unit tests, provider contract tests, API compatibility check, migrations test, container smoke test, dependency/license scan, CodeQL, Trivy, secret scan, SBOM, signed image, changelog, upgrade/rollback notes. The repository workflows automate the build/test, generated-client drift, migration-model drift, image smoke, CodeQL, Trivy, SBOM, provenance, and signing gates; release notes, compatibility evidence, license review, and upgrade/rollback notes remain explicit release-manager approvals. Database rollback is restore/forward-fix, not down migrations.
