# Provider Architecture and Compatibility

Operational diagnosis by stable outcome and provider family is in `docs/PROVIDER_TROUBLESHOOTING.md`.

## Contract

Providers expose capability slices: `probe`, `library`, `missing`, `queue`, `history`, `search`, `grab`, `tasks`, `health`, `download-client`, `media-server`, `requests`, and `notifications`. Capability discovery controls UI and API actions. Every call has timeout, cancellation, typed errors, redacted logging, and rate-limit metadata.

Credentials are referenced through a secret handle. Adapters never persist secrets or use global mutable HTTP clients. Provider DTOs stay inside adapter assemblies; mapping produces versioned canonical records.

## Compatibility matrix

All rows without a contract state remain planned, not supported. Configuring an instance kind never changes that status. Arr tasks, queue mutations, interactive release search/grab, and other mutation slices remain unavailable until their contract implementations land. A successful version-checked probe persists evidence only for the contract-tested capabilities above.

The checked-in Sonarr [`4.0.1.1168`](https://github.com/Sonarr/Sonarr/blob/v4.0.1.1168/src/Sonarr.Api.V3/openapi.json)/[`4.0.19.2979`](https://github.com/Sonarr/Sonarr/blob/v4.0.19.2979/src/Sonarr.Api.V3/openapi.json) and Radarr [`5.3.1.8438`](https://github.com/Radarr/Radarr/blob/v5.3.1.8438/src/Radarr.Api.V3/openapi.json)/[`6.3.0.10514`](https://github.com/Radarr/Radarr/blob/v6.3.0.10514/src/Radarr.Api.V3/openapi.json) fixtures are synthetic-minimal, redacted projections of the official tagged OpenAPI contracts. They cover status, health, Sonarr series/seasons/episodes, and Radarr movies. Newer fixtures include unknown fields/statuses, which normalize fail-soft to `unknown` while selected upstream values remain in non-secret provider data.

Sonarr catalog synchronization reads `GET /api/v3/series` and the documented `GET /api/v3/episode?seriesId=...` slice for each series; Radarr reads `GET /api/v3/movie`. Neither contract exposes a reliable library delta token. ArrControl therefore performs an idempotent local snapshot diff: it inserts new natural keys, updates only changed fingerprints, removes keys absent from a complete successful snapshot, and advances the `catalog` checkpoint only after job completion. A failed or malformed fetch never replaces the last-known catalog.

Missing search uses only the official command boundary: Sonarr `EpisodeSearch` with bounded `episodeIds`, and Radarr `MoviesSearch` with bounded `movieIds`, posted to `/api/v3/command`. A successful queued command ID is retained only in the target's bounded operation result. This capability does not include release selection, grabbing, broad series searches, or searching non-missing/unmonitored items.

Lidarr uses the official v1 artist/album, queue/history, health/status, and `AlbumSearch` contracts for [`2.9.6.4552`](https://github.com/Lidarr/Lidarr/blob/v2.9.6.4552/src/Lidarr.Api.V1/openapi.json) and [`3.1.3.4975`](https://github.com/Lidarr/Lidarr/blob/v3.1.3.4975/src/Lidarr.Api.V1/openapi.json). Readarr uses the corresponding v1 author/book resources and `BookSearch` command for [`0.4.17.2801`](https://github.com/Readarr/Readarr/blob/v0.4.17.2801/src/Readarr.Api.V1/openapi.json) and [`0.4.18.2805`](https://github.com/Readarr/Readarr/blob/v0.4.18.2805/src/Readarr.Api.V1/openapi.json). Support is intentionally limited to those Readarr 0.4 contracts.

Whisparr changed its library model between major versions. The [`2.2.0-release.108`](https://github.com/Whisparr/Whisparr/blob/v2.2.0-release.108/src/Whisparr.Api.V3/openapi.json) contract is normalized from series/episodes and searches missing `episodeIds`; [`3.1.0.2116`](https://github.com/Whisparr/Whisparr/blob/v3.1.0.2116/src/Whisparr.Api.V3/openapi.json) is normalized from movies and searches missing `movieIds`. A v2 catalog probe receives the contract-defined not-found response for the v3-only movie route before selecting the series path. Unknown upstream enum fields normalize to `unknown` in both models.

Prowlarr support follows the official tagged v1 contracts for [`2.4.0.5397`](https://github.com/Prowlarr/Prowlarr/blob/v2.4.0.5397/src/Prowlarr.Api.V1/openapi.json) and [`2.5.1.5464`](https://github.com/Prowlarr/Prowlarr/blob/v2.5.1.5464/src/Prowlarr.Api.V1/openapi.json). The indexer projection deliberately excludes configuration fields and URLs; history retains only the indexer display name, stable event state, event time, and download correlation ID. Bazarr support is derived from official [`1.5.3`](https://github.com/morpheus65535/bazarr/tree/v1.5.3/bazarr/api) and [`1.6.0`](https://github.com/morpheus65535/bazarr/tree/v1.6.0/bazarr/api) API source contracts. Bazarr health object values can contain filesystem paths or private profile names, so ArrControl replaces them with a stable digest source and retains only the bounded issue text. Detail-history display timestamps are locale-relative; ArrControl therefore exposes only the exact `yyyy-MM-dd` daily subtitle-activity statistics rather than inventing event timestamps.

SABnzbd support follows the official [`4.5.3`](https://github.com/sabnzbd/sabnzbd/tree/4.5.3) and [`5.0.4`](https://github.com/sabnzbd/sabnzbd/tree/5.0.4) releases plus the [5.0 API contract](https://sabnzbd.org/wiki/configuration/5.0/api). It reads `/api` version, queue, and history modes and uses the documented queue pause/resume/delete and history retry actions. SABnzbd requires `apikey` in the query string; this is the sole supported credential-query exception, is constructed only inside the adapter, and is never logged. NZBGet follows the official JSON-RPC contracts for [`v25.4`](https://github.com/nzbgetcom/nzbget/blob/v25.4/docs/api/API.md) and [`v26.2`](https://github.com/nzbgetcom/nzbget/blob/v26.2/docs/api/API.md), using HTTP Basic authentication and `version`, `status`, `listgroups`, `history`, and `editqueue` methods.

qBittorrent support targets the official WebUI API for [`release-5.1.4`](https://github.com/qbittorrent/qBittorrent/wiki/WebUI-API-%28qBittorrent-5.0%29) and [`release-5.2.3`](https://github.com/qbittorrent/qBittorrent/wiki/WebUI-API-%28qBittorrent-5.0%29). It performs the Origin/Referer-bound login, retains only the bounded SID cookie for one call, and uses the v5 `start`, `stop`, and `delete` endpoints. Transmission follows the official RPC contracts for [`4.0.6`](https://github.com/transmission/transmission/blob/4.0.6/docs/rpc-spec.md) and [`4.1.3`](https://github.com/transmission/transmission/blob/4.1.3/docs/rpc-spec.md); ArrControl deliberately uses the retained legacy camel-case request form across both 4.x versions and retries the documented 409 response once with `X-Transmission-Session-Id`. Deluge follows the official Web JSON-RPC flows for [`deluge-2.1.1`](https://github.com/deluge-torrent/deluge/blob/deluge-2.1.1/docs/source/devguide/how-to/curl-jsonrpc.md) and [`deluge-2.2.0`](https://github.com/deluge-torrent/deluge/blob/deluge-2.2.0/docs/source/devguide/how-to/curl-jsonrpc.md), authenticates with its Web password, selects the first configured daemon host, connects when necessary, and calls only bounded core methods.

Torrent adapters expose completed torrents that remain present in the client's current torrent list as normalized history. They do not claim a retained client history after a torrent has been removed. Retry is intentionally unsupported for qBittorrent, Transmission, and Deluge because their contracts do not provide an equivalent bounded retry-history action.

Plex support uses the official [Plex Media Server API 1.0 contract](https://developer.plex.tv/pms/), which is available from PMS 1.41.9, with fixtures for `1.41.9.9961` and the official Docker tag [`1.43.2.10687-563d026ea`](https://hub.docker.com/r/plexinc/pms-docker/tags). Calls use JSON, the `X-Plex-Token` header, a per-instance client identifier, `/identity`, `/library/sections`, and `/status/sessions`. Jellyfin support is derived from the official tagged controller contracts for [`v10.10.7`](https://github.com/jellyfin/jellyfin/tree/v10.10.7/Jellyfin.Api/Controllers) and [`v10.11.10`](https://github.com/jellyfin/jellyfin/tree/v10.11.10/Jellyfin.Api/Controllers). Emby support follows the official [`4.8.11.0`](https://github.com/MediaBrowser/Emby.Releases/releases/tag/4.8.11.0) and [`4.9.3.0`](https://github.com/MediaBrowser/Emby.Releases/releases/tag/4.9.3.0) releases plus its [`/System/Info`](https://dev.emby.media/reference/RestAPI/SystemService/getSystemInfo.html), [`/Items/Counts`](https://dev.emby.media/reference/RestAPI/LibraryService/getItemsCounts.html), and [`/Sessions`](https://dev.emby.media/reference/RestAPI/SessionsService/getSessions.html) contracts. Jellyfin uses the same three unprefixed routes; Emby uses the documented `/emby` API prefix. Both use `X-Emby-Token` API-key authentication.

Media-server normalization is deliberately aggregate-only. Plex library section names, Jellyfin/Emby server names, playback titles, usernames, device names, client addresses, and every media/transcode path are discarded inside the adapter. The canonical snapshot retains only library/media-type counts and counts of active, playing, paused, buffering, transcoding, direct-stream, direct-play, or unknown-delivery sessions. It is a point-in-time availability/activity contract, not playback history, remote control, library mutation, or a claim that the media files themselves were opened and verified.

Overseerr support follows the official tagged [`v1.34.0`](https://github.com/sct/overseerr/tree/v1.34.0/server/routes) and archived final [`v1.35.0`](https://github.com/sct/overseerr/tree/v1.35.0/server/routes) request/status routes. Jellyseerr support follows [`v2.7.3`](https://github.com/seerr-team/seerr/tree/v2.7.3/server/routes) and the renamed Seerr [`v3.3.0`](https://github.com/seerr-team/seerr/tree/v3.3.0/server/routes) contract. Both use `X-API-Key` and bounded `/api/v1/request` results. Ombi follows its tagged [`v4.47.1`](https://github.com/Ombi-app/Ombi/tree/v4.47.1/src/Ombi/Controllers/V1) and [`v4.48.0`](https://github.com/Ombi-app/Ombi/tree/v4.48.0/src/Ombi/Controllers/V1) V1 controllers with the `ApiKey` header and separate movie/TV inventories. Request titles, users, aliases, rejection reasons, root folders, and raw payloads are discarded. Only stable provider keys, normalized status/type, request time, and TMDB/TVDB/IMDb correlation IDs leave the adapter; unknown statuses fail soft to `unknown`.

Recyclarr publishes a CLI and scheduled container, not a remotely addressable HTTP service. ArrControl therefore does not invent a `recyclarr` URL kind or send credentials to an unofficial wrapper. Its source-level task adapter follows the official [`sync`](https://recyclarr.dev/cli/sync/) contract for Recyclarr 7/8: `--version`, optional `sonarr|radarr`, repeated validated `--instance`, and optional `--preview`. It uses `ProcessStartInfo.ArgumentList` with no shell, absolute executable/configuration paths, a cleared child environment containing only Recyclarr directories and `NO_COLOR`, bounded output/time, cancellation with process-tree termination, and redacted results. Preview is non-mutating; non-preview sync is deliberately an explicit call. Packaging or mounting the trusted native executable/configuration remains an operator deployment choice and does not make Recyclarr an ArrControl HTTP instance.

Notification adapters implement the common bounded `NotificationMessage` contract and discard provider responses after validating success. Generic webhooks send versioned JSON and optionally sign its exact UTF-8 bytes with HMAC-SHA-256. Discord and Slack use their official incoming-webhook JSON; Teams uses the current Workflows webhook Adaptive Card envelope rather than the retiring Office 365 Connector contract. Telegram uses Bot API `sendMessage`, ntfy publishes JSON to a topic with optional bearer authentication, Gotify posts JSON with `X-Gotify-Key`, and Pushover posts the required application/user keys in JSON to `/1/messages.json`. Webhook URLs and Telegram bot-token paths are treated as secrets and never rendered. Email uses RFC 8314-style implicit-TLS submission (`smtps`, normally port 465), pinned DNS answers, certificate hostname validation, SMTP AUTH PLAIN inside TLS, bounded responses, sanitized addresses/headers, and dot-stuffed UTF-8 text. This slice is the outbound adapter boundary; notification routing/preferences and durable delivery selection are separate application concerns and no destination is contacted automatically.

“Supported” HTTP providers in release notes require fixtures for at least two upstream versions, authentication failure coverage, unknown-field tolerance, rate-limit handling, and a live opt-in smoke test. Non-HTTP adapters require equivalent boundary, injection, timeout, cancellation, redaction, and version evidence.

Read-only live smoke tests are opt-in through matching `ARRCONTROL_LIVE_<PROVIDER>_BASE_URL` plus the adapter-required `API_KEY`, `USERNAME`, and/or `PASSWORD` variables. They cover every contract-supported HTTP provider listed above; private targets additionally require the matching `ALLOW_PRIVATE_NETWORK=true`. `DISABLE_TLS_VERIFICATION=true` is available only for an explicitly accepted test exception. The tests never read the development `.env`, mutate an upstream, or print credentials. Recyclarr has no live sync smoke because even its preview contacts configured services and its normal command mutates them.

## Error taxonomy

`credential_missing`, `unreachable`, `tls_error`, `unauthorized`, `forbidden`, `rate_limited`, `unsupported_version`, `invalid_response`, `upstream_conflict`, `not_found`, `timeout`, and `unknown`. Raw upstream messages are diagnostic fields; localized user messages derive from stable codes.

## Plugin evolution

In-process signed assemblies are not loaded from arbitrary paths in v1. The first extension mechanism is source-level provider packages. A v2 provider host may use a versioned gRPC contract, isolated container, declared permissions, health handshake, and compatibility negotiation.
